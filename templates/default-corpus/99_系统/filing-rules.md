# Filing Rules

> Agent 归档决策的规则表。读完这份，你就知道任何一条内容该放进哪个目录。

## 铁律

> **The PRIMARY SUBJECT of the content determines where it goes.**
> **Not the format, not the source, not the skill that's running.**

问自己一个问题：**"这条内容到底在讲谁/什么？"**

答案决定目录。一条内容可以有多个主语，那就**每个主语都处理一遍**（一份原文 + 多个 timeline append）。

## 路由表

| 内容类型 | 主语判断 | 目标目录 |
|---|---|---|
| 文章 | 文章作为资料本身 | `60_来源/文章/` |
| 文章 | 里面讨论的概念 | `30_概念/<概念名>.md` 追加 timeline |
| 文章 | 作者（值得追踪） | `10_人物/<作者名>.md` 追加 timeline |
| 会议 | 会议作为资料本身 | `60_来源/会议纪要/` |
| 会议 | 会上讨论的项目 | `20_项目/<项目名>.md` 追加 timeline |
| 会议 | 会上认识的人 | `10_人物/<人名>.md`（过 Notability gate） |
| 录音转写 | raw 文本 | `70_录音/YYYY-MM-DD/` |
| 录音转写 | 结论性内容 | 按主语移到 10/20/30 |
| 自己的想法 | 没想清楚 | `_工作台/00_收件/` |
| 自己的想法 | 关于某人/某项目 | 对应目录追加 timeline |
| 日常情绪 | 日记性质 | `00_每日/YYYY-MM-DD.md` |
| 写作草稿 | 正在写的文章 | `80_写作/` |
| 方法 / SOP | 可复用工具操作 | `50_方法/` |

## sources 原则

`60_来源/` **严格只读**。里面只放原始数据快照（文章全文、会议纪要、书籍摘抄、公众号剪藏）。

**任何有明确主语的分析都必须移走**。比如"这篇文章讲 RAG 的好处" —— RAG 是主语，分析放 `30_概念/RAG.md` 的 Timeline 里，**不要**在 `60_来源/` 里夹带评论。

## Notability gate

不是所有实体都值得建独立页面。建新页前先问：

> **"下次我会主动引用这个实体吗？"**

- **会** → 建新页（frontmatter + Compiled Truth + Timeline 首条）
- **不会** → 找最近相关页追加一条 timeline

例子：
- 地铁上偶遇聊了两句的陌生人 → 不建页，写进日记
- 面试官 → 建页（会反复回顾）
- 随手看的一篇博客作者 → 不建页，在文章条目里提一下
- 长期订阅的 newsletter 作者 → 建页

## 至少一条反向链接

**任何 ingest 都必须至少建一条反向链接**。

- 新建的页面要在 `related_*` 或 Timeline 里链到来源
- 被链的目标页也要在 Timeline 留一条反向记录
- 没有链接的信息等于没有

孤岛页面会被 `wiki lint` 报告出来。

## 当规则打架时

规则之间打架的场景：

**"这篇文章既讨论概念 A，也介绍了作者 B，还涉及项目 C，该放哪？"**

答：**三份都做**：
1. 原文 → `60_来源/文章/`
2. 在 `30_概念/A.md` 的 Timeline append 一条
3. 在 `10_人物/B.md` 的 Timeline append 一条
4. 在 `20_项目/C.md` 的 Timeline append 一条

四条都链回来源。这不是冗余，是多视角索引。

## 拿不定主意怎么办

先扔进 `_工作台/30_待整理/`，加一条 frontmatter 注释"需要人类决定主语"。lorekit 的 `wiki lint` 会定期提醒你清这个目录。

**永远不要**把拿不定主意的东西扔进 `60_来源/` 装没看见——那是对 sources 目录的污染。
